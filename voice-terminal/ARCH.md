## 二、硬件方案

### 方案 A：低成本（~150 元）— ESP32-S3

| 器件 | 型号 | 价格 | 说明 |
|:----|:----|:---:|:-----|
| 主控 | **ESP32-S3** N16R8 | ¥45 | 双核 240MHz，16MB Flash，8MB PSRAM |
| 麦克风 | **INMP441** (I2S) | ¥8 | 单麦，全向，-26dBFS |
| 功放 | **MAX98357** (I2S) | ¥15 | 3.2W @ 4Ω，D 类功放 |
| 喇叭 | 3W 4Ω 钕磁喇叭 | ¥10 | 直径 36mm，厚度 7mm |
| LED | WS2812 环形灯板 | ¥12 | 8 颗，做唤醒指示灯环 |
| 电池 | 18650 × 1 | ¥15 | 2600mAh，约 6h 续航 |
| 充电 | TP4056 模块 | ¥5 | 5V 1A 充电 |
| 按键 | K2 贴片 ×2 | ¥2 | 静音键 + 自定义键 |
| 外壳 | 3D 打印 PLA/树脂 | ¥30 | 需设计开模 |
| PCB | 打样 10×10cm | ¥20 | 嘉立创 5 片 |
| **合计** | | **~¥160** | |

**PCB 尺寸：** 60×60mm，四层板
**功耗：** 待机 150mW（ESP32 Deep Sleep + 唤醒词检测），对话中 1.2W

### 方案 B：高配（~600 元）— 树莓派 Zero 2W

| 器件 | 价格 | 理由 |
|:----|:---:|:-----|
| Raspberry Pi Zero 2W | ¥120 | 1GHz 四核，512MB RAM |
| ReSpeaker 2-Mic HAT | ¥80 | 双麦降噪，XMOS 音频处理 |
| 3W 喇叭 + 功放 | ¥25 | |
| 电池管理 + 18650 | ¥30 | |
| 外壳 | ¥40 | |
| **合计** | **~¥300** | |

**优势：** 板载跑 Whisper tiny 成为可能，不依赖服务器 ASR
**劣势：** 体积是 ESP32 方案的 3 倍，功耗 ~2.5W

### 推荐：先用 ESP32-S3 方案 A

理由：
- 成本低，打样快
- 唤醒词、音频流、播放已足够
- 所有 AI 处理在服务端（你 PC），ESP32 只做 I/O

## 三、软件组件

### 3.1 网关服务（新增，~500 行 Python）

```
voice-gateway/
├── gateway.py          # WebSocket 服务器，连接管理
├── router.py           # 路由引擎（模型选择策略）
├── asr.py              # Whisper 调用封装
├── tts_engine.py       # TTS 引擎选择 + 调用
├── session.py          # 对话上下文管理
├── config.yaml         # 模型路由规则配置
└── speaker_profiles/   # 克隆音色存储
    └── user_ref.wav    # 用户的参考音频
```

### 3.2 硬件固件（ESP-IDF / Arduino）

```
esp32-firmware/
├── main/
│   ├── main.c           # 主循环
│   ├── mic_i2s.c        # 麦克风 I2S 驱动
│   ├── speaker_i2s.c    # 喇叭 I2S 驱动
│   ├── wake_word.c      # Porcupine 唤醒词
│   ├── websocket.c      # WebSocket 客户端
│   ├── audio_codec.c    # opus 编解码
│   └── led_ring.c       # WS2812 灯效
├── sdkconfig
└── CMakeLists.txt
```

### 3.3 现有软件复用

| 已有组件 | 位置 | 在项目中的作用 |
|:--------|:----|:-------------|
| Whisper tiny | local-voice skill | ASR |
| Coder-Next | 8082 service | 本地编码/推理 LLM |
| MOSS-TTS | ~/MOSS-TTS/ | 声音克隆 TTS |
| Edge-TTS | 系统 pip | 快速 TTS |
| Hermes | systemd user | 通知、控制、技能联动 |

### 3.4 路由策略（`config.yaml`）

```yaml
routers:
  - name: "编程"
    match_keywords: ["写代码", "bug", "函数", "重构", "git", "push", "PR"]
    model: "local://coder-next:8082"
    
  - name: "系统操作"
    match_keywords: ["帮我", "执行", "安装", "删除文件", "关机"]
    model: "hermes://execute"
    
  - name: "隐私"
    match_keywords: ["我的密码", "我的私钥", "个人文件"]
    model: "local://35b:8080"
    
  - name: "闲聊/常识"
    match_weight: 0.8  # 无关键词匹配时的兜底
    model: "cloud://deepseek"  # 或豆包 API
    
  - name: "自定义模型"
    match_keywords: ["用豆包", "换ChatGPT", "切本地"]
    action: "switch_model"  # 热切换会话模型
```

## 四、对话体验流程

```
1. "小智小智" (唤醒词)
   → ESP32 检测到，灯环亮蓝色
   → 开始录音

2. "帮我写一个 Python 快速排序"
   → ESP32 压缩音频 → WebSocket 推流
   → 服务器 Whisper 识别 → "帮我写一个 Python 快速排序"
   → 路由引擎匹配关键词 "写代码" → Coder-Next
   → Coder-Next 生成回复
   → TTS 引擎: 生成较短 → Edge-TTS (0.5s)
   → 音频流推回 ESP32 → 喇叭播放
   → 灯环呼吸灯 → 完成后熄灭

3. "用我的声音说话"
   → 路由引擎检测到"我的声音" → 触发克隆流程
   → "好的，请说一段 5 秒的话"
   → 录音 5 秒 → 存为 speaker_profiles/user_ref.wav
   → 提示: "音色已保存"

4. "今天天气怎么样"
   → 路由引擎: 无编程/系统/隐私关键词 → 闲聊 → DeepSeek API
   → LLM 回答 → Edge-TTS → 播放
```

## 五、延迟分布

```
唤醒词检测:   ~80ms   (ESP32 本地)
录音推流:     ~300ms  (1 秒语音缓冲)
Whisper:      ~300ms  (GPU ROCm)
路由判断:     ~5ms
LLM:          ~2-4s   (Coder-Next ~39 tok/s, 50-100 tok 回答)
TTS:          ~500ms  (Edge-TTS) / ~2.5s (MOSS-TTS)
网络传输:     ~50ms
播放缓冲:     ~100ms
──────────────────────────────────
总计 Edge:    ~3.3s
总计 克隆:    ~5.5s  (MOSS-TTS 占大头)
```

优化点：
- **流式 TTS**：不等 LLM 出完再 TTS，边出边 TTS → 可省 ~1s
- **预缓冲**：唤醒后预载 Whisper 模型 → 省 ~100ms

## 六、版本规划

### V0 — 软件原型（2-3 天）
```
PC 麦克风 → Whisper → 路由 → LLM → TTS → PC 喇叭
```
验证链路可行性和延迟

### V1 — ESP32 原型（1-2 周）
```
ESP32 录音 → WiFi → 服务器 → 处理 → WiFi → ESP32 播放
```
打 PCB + 3D 打印外壳 + 固件

### V2 — 完整产品（3-4 周）
```
+ 唤醒词
+ LED 灯效
+ 热切换模型
+ 声音克隆管理
+ Hermes 联动
```

## 七、开源/商业化

- **软件全部开源**（网关 + 固件 + 外壳 STL）
- **硬件 BOM + PCB 公开**
- **用户只需准备：** ESP32-S3 开发板 + INMP441 + MAX98357 + 喇叭 ≈ ¥50 可玩
- **差异化：** 市场上有豆包音箱、小爱音箱，但没有"任意模型可切换 + 声音克隆"的开放硬件方案
